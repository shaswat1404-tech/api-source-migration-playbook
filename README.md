# API Source Migration Playbook
### Migrating a Production Data Platform Across API Providers | Senior Housing / Property Management Industry

> A team playbook documenting the end-to-end approach for migrating a production operational analytics platform from one property management API provider to another — covering discovery, gap analysis, field mapping, historical backfill, platform rebuild, phased validation, and cutover.

---

## Overview

This playbook documents the approach taken by a cross-functional team to migrate a production data platform from **ManageAmerica (MA)** to **Rent Manager (RM)** as the primary operational data source. The migration involved not just an API swap but a full re-evaluation of data availability, granularity, load strategy, and historical data continuity.

**Team involved:** Business Analyst, Data Engineering Solution Architect, Data Engineers, QA, Power BI Developer, Power BI Solution Architect, Scrum Master, Project Manager

**Scale of change:**
- MA: ~5–6 endpoints, all full load on every run
- RM: 35+ endpoints, mix of incremental and full load strategies
- Historical data continuity required across all reporting breakdowns
- Hard cutover window: 3 days
- Dashboard go-live: planned 1 week, extended to 3–4 weeks due to API team dependencies

---

## Why This Migration Was Complex

On the surface, this looks like swapping one REST API for another. In practice it was significantly more involved:

- **RM required far more endpoints than MA** to cover equivalent data — 35+ vs 5–6, each with its own schema, pagination, rate limits, and load behaviour
- **Granularity was not guaranteed** — MA data supported breakdowns by region, division, property, and portfolio; RM had to be assessed endpoint-by-endpoint for equivalent granularity support
- **No standardised historical data handover** — some history was available via RM API full loads, some required the client to generate ID mapping files, and some required custom workarounds
- **Field-level mapping was manual and time-consuming** — no automated schema translation between MA and RM
- **RM API team dependencies** — some required data points were not yet available in RM, requiring escalation, workaround design, or deferral decisions with business stakeholders

## Phase 1 — Discovery & Gap Analysis

### Technical Document Review
A full technical analysis was conducted against the existing MA integration documentation — covering every endpoint in use, the data it returned, the load strategy, the downstream tables it fed, and the reporting breakdowns it supported.

### Stakeholder Engagement with RM API Team
A series of calls were held with the RM API team to:
- Present our data requirements in full
- Understand what RM already supported natively
- Identify gaps — what RM could not provide, what required development time on their end, and what needed a workaround on ours
- Agree timelines for RM-side deliverables

### Granularity Validation
Granularity was validated not just at date level but across all required reporting dimensions:

| Dimension | Verified |
|---|---|
| Region | ✓ |
| Division | ✓ |
| Portfolio | ✓ |
| Property | ✓ |
| Other breakdowns | Case by case |

Where RM did not natively support a required breakdown, options were assessed: request the API admin to add it, design a workaround using available fields, or escalate to business stakeholders for a decision.

### Load Strategy Assessment
Unlike MA (all full load), RM endpoints required a deliberate load strategy per endpoint:

| Load type | Applied when |
|---|---|
| Incremental | Endpoint supports change tracking / timestamps |
| Full load | Endpoint does not support incremental; data volume allows full reload |

This was documented per endpoint before any development began.

### Ongoing Stakeholder Communication
Throughout discovery, regular calls were held with client and business stakeholders covering:
- Status updates and milestone tracking
- Workaround discussions and decisions
- Change communications (where RM data differed from MA expectations)
- Cutover planning and dashboard release dates
- All decisions and outcomes documented on the go

## Phase 2 — Field Mapping

Field mapping between MA and RM was **manual and time-consuming** — there was no automated schema translation between the two providers.

Key challenges:
- Field names, data types, and value formats differed significantly between MA and RM
- Some MA fields had no direct RM equivalent and required a derived or workaround approach
- Entity identifiers (IDs) changed between systems — MA IDs and RM IDs are not the same
- Mapping tables either came from the RM API directly or from the client's MDM system — they were not generated by the engineering team

The output of this phase was a complete field-level mapping covering every data point required downstream, with decisions documented for fields that could not be mapped directly.

---

## Phase 3 — Historical Data & Backfill

Historical data continuity was one of the most complex aspects of this migration. Three distinct scenarios required three different approaches:

### Scenario A — History Available via RM API
Some historical data was loaded into RM by the RM team ahead of go-live. This data could be pulled directly via full-load API calls and processed through the standard pipeline. No custom logic required beyond the full-load ingestion path.

### Scenario B — Client-Provided ID Mapping
For records where MA entity IDs did not exist in RM, the client generated mapping files that translated old MA IDs to new RM IDs. These mapping files were ingested and used to join historical MA data to the new RM schema — inserting correctly keyed historical records into the platform.

### Scenario C — Engineering Workarounds
For cases where neither API history nor client mappings were available, workarounds were designed by the team:
- **Negative IDs** assigned to historical records to make them identifiable and distinguishable from live RM data
- **`is_source_rm_api` flag** added to all records — `True` for live RM API data, `False` for backfilled historical records — enabling clear source identification during debugging and validation
- These flags proved critical during integration testing and go-live, where data discrepancies needed to be traced back to their source quickly

> If RM had provided complete historical data through its APIs, a significant portion of this engineering effort would not have been required. The workarounds above were pragmatic solutions to a real constraint.

## Phase 5 — Phased Validation

Validation was structured in three phases, each building on the previous:

### Phase 1 — Static Historical Validation
The MA API data (up to a fixed date) was provided to the RM team, who loaded it into RM. A static comparison was then performed between the MA source data and what RM returned for the same period — validating that historical records were correctly represented in the new system before any live data was involved.

### Phase 2 — End-to-End Pipeline Validation
With historical data in place, the full pipeline was validated end-to-end:
- RM API → Blob → Databricks Bronze → Silver → Gold → Power BI
- Output values at the Gold/report layer compared against known-good MA-based outputs for the same period
- QA team and PBI team involved at the reporting layer
- Data discrepancies traced using the `is_source_rm_api` flag and negative ID markers from the backfill

### Phase 3 — Live Data Validation
Once the pipeline was stable on historical data, live data testing began:
- Real operational changes made in RootsMG were tracked through the full pipeline
- Verified that changes captured by RM APIs were correctly processed by Databricks ELT and reflected accurately in dashboards
- Edge cases and timing issues identified and resolved before cutover

---

## Cutover

**Hard cutover window: 3 days**
During this period, RootsMG was required to either pause recording changes or manually note them for later entry into RM — ensuring no data was lost in the transition gap.

**Dashboard go-live:** Planned for 1 week post-cutover. Extended to 3–4 weeks due to outstanding issues and delays from the RM API team on certain endpoints.

**Key cutover activities:**
- Final data reconciliation between MA and RM outputs
- Confirmation that all RM endpoints were stable and returning expected data
- Sign-off from business stakeholders, QA, and PBI teams
- Decommission of MA ingestion pipelines post-stabilisation

## Key Lessons

**API parity is rarely 1:1.** 5–6 MA endpoints expanding to 35+ RM endpoints is not unusual when switching providers — plan for it explicitly, not as a surprise mid-project.

**Granularity must be validated before field mapping.** If the new API cannot return data at the required breakdown level, no amount of field mapping fixes that. Granularity validation comes first.

**Historical data is always the hard part.** Budget significant time for it. The three-scenario approach (API history / client mapping / engineering workaround) is a realistic framework for most API migrations — rarely does one approach cover everything.

**Flags and identifiers are debugging lifelines.** The `is_source_rm_api` flag and negative ID convention paid for themselves many times over during go-live debugging. Build traceability into the data model from day one.

**API team dependencies are a project risk.** Delays from the RM API team directly pushed the dashboard go-live from 1 week to 3–4 weeks. Identify these dependencies early, escalate often, and have contingency plans.

**Document decisions as they happen.** With a team of this size across multiple workstreams, decisions made on a Tuesday call are forgotten by Thursday. A running decision log is not optional.
